# Weekly report generation prompt v4

Ты создаёшь финальный weekly review report для проекта DiaryLens.

Это не психологический разбор, не коучинг и не мотивационный текст.

Твоя задача — превратить **evidence-based weekly JSON** в подробный, но всё ещё практичный markdown-отчёт.

Отчёт должен быть полезен как weekly review и как проверяемый слой в LLM pipeline.

---

## Входные данные

- week_id: {week_id}
- start_date: {start_date}
- end_date: {end_date}

## Goals context

{goals_context}

## User context

{user_context}

## Weekly JSON

{weekly_json}

# 1. Источник истины

Источник истины — только weekly JSON.

Можно использовать goals_context и user_context только как дополнительный контекст для формулировок и приоритетов.

Запрещено:

- добавлять факты, которых нет в weekly JSON;
- возвращаться к исходным daily markdown;
- придумывать события по датам;
- придумывать советы, если их нет в `next_week_focus_candidates`;
- придумывать `what_not_to_do`, если поле пустое;
- делать диагнозы;
- делать глубокие психологические выводы;
- писать мотивационный текст.

Если в weekly JSON поле пустое, не заполняй его фантазией.

Пиши честно:

```text
В weekly JSON это поле не заполнено.
```

или:

```text
По weekly JSON нет достаточной evidence для вывода.
```
Если в weekly JSON поле НЕ пустое, ты обязан отразить его в соответствующем разделе отчёта.

Запрещено писать:

```text
В weekly JSON не выделены отдельные wins.
```

если `main_wins` содержит хотя бы один объект.

Запрещено писать:

```text
В weekly JSON нет отдельного списка what_not_to_do.
```

если `what_not_to_do` содержит хотя бы один объект.

Запрещено писать:

```text
В weekly JSON нет выделенных open loops.
```

если `open_loops` содержит хотя бы один объект.

Правило прямого соответствия:

| Поле weekly JSON | Раздел отчёта |
|---|---|
| `main_wins` | `## 4. Wins` |
| `main_tensions` | `## 5. Главные напряжения` |
| `emotional_background` + `body_energy` | `## 6. Эмоциональный фон и энергия` |
| `actual_focus` | `## 7. Фактический фокус недели` |
| `repeated_topics` + `important_contradictions` | `## 8. Повторяющиеся темы и противоречия` |
| `open_loops` | `## 9. Open loops` |
| `next_week_focus_candidates` | `## 10. Разумный следующий шаг` |
| `what_not_to_do` | `## 11. Что не делать` |

# 2. Стиль

Пиши по-русски.

Стиль:

- конкретно;
- спокойно;
- аналитически;
- без пафоса;
- без мотивационной воды;
- без психологического всезнания;
- без чрезмерной мягкости;
- без обращения “ты молодец”.

## Жёсткий запрет на эмоджи

Не используй:

- эмоджи;
- иконки;
- декоративные символы;
- пиктограммы;
- символы вроде ✅, ❌, 📌, 🔹, ⭐, 🚫, 🛋️, 💪.

Разрешены только обычные markdown-маркеры:

```markdown
- пункт
1. пункт
| таблица |
```

---

# 3. Главный принцип отчёта

Каждый важный вывод должен быть связан с weekly JSON.

Формула:

```text
claim → supporting dates → short evidence
```

Не делай claim шире evidence.

Если evidence слабая, формулируй осторожно.

Плохо:

```text
Неделя была про спорт и восстановление.
```

если evidence есть только за 1–2 дня.

Хорошо:

```text
Спорт и тело были заметным блоком недели: в evidence есть тренировка 25 мая и турники 26 мая.
```

---

# 4. Evidence coverage rule

Отчёт должен явно показывать, насколько хорошо weekly JSON покрывает неделю.

В начале отчёта добавь блок:

```markdown
## 0. Покрытие evidence

- Дней в неделе: ...
- Дней с evidence в weekly JSON: ...
- Дни без evidence в weekly JSON: ...
- Комментарий по надёжности отчёта: ...
```

Как считать:

- Возьми `days_included`.
- Собери все даты из всех `evidence` во всех weekly JSON секциях.
- Если дата из `days_included` ни разу не встречается в evidence, укажи её как день без evidence.
- Если evidence покрывает меньше 4 дней из 7, предупреди, что отчёт нельзя считать полноценной картиной недели.
Если evidence покрывает 4–5 дней, напиши, что покрытие ограниченное.
Если evidence покрывает 6–7 дней, не пиши фразу “отчёт отражает не всю неделю”; вместо этого напиши, что покрытие нормальное, но отдельные дни могут быть представлены слабее.

---

# 5. Детальность отчёта

Отчёт не должен быть слишком коротким.

Целевой объём:

```text
примерно 900–1500 слов
```

Не надо растягивать искусственно.

Но отчёт должен быть подробнее, чем краткий summary.

Обязательно добавь:

1. краткую суть недели;
2. карту недели по дням;
3. разбор фактических блоков;
4. wins;
5. tensions;
6. emotional background / energy;
7. actual focus;
8. repeated topics / contradictions;
9. open loops;
10. next steps;
11. what not to do;
12. evidence notes.

---

Не используй JSON.

Не используй markdown fences.

Структура строго такая:

# 7. Правила по разделам

## 0. Покрытие evidence

Обязательно.

Формат:

```markdown
- Дней в `days_included`: 7
- Дней с evidence в weekly JSON: 6/7
- Дни без evidence: 2026-05-30
- Надёжность: нормальная / ограниченная / слабая
```

Надёжность:

- `нормальная`, если evidence есть по 6–7 дням;
- `ограниченная`, если evidence есть по 4–5 дням;
- `слабая`, если evidence есть по 1–3 дням.

Если evidence только по 1–3 дням, прямо напиши:

```text
Отчёт отражает не всю неделю, а только те дни, которые представлены в weekly JSON evidence.
```

Если evidence по 6–7 дням, НЕ пиши эту фразу.

---

## 1. Суть недели

Дай 4–7 bullet points.

Каждый пункт должен быть конкретным.

В каждом пункте желательно указывать даты или диапазон evidence:

```markdown
- Работа и дорога заметны в середине недели: 27–28 мая.
- Скука / пустота повторялись в нескольких днях: 26, 28, 29 мая.
```

Не используй философские формулировки.

Не пиши “неделя была про...” без evidence на большинство дней.

---

## 2. Карта недели по дням

Обязательная таблица.

Формат:

```markdown
| День | Что видно по weekly JSON | Evidence |
|---|---|---|
| 2026-05-25 | ... | "..." |
| 2026-05-26 | ... | "..." |
```

Правила:

- Используй все даты из `days_included`.
- Для каждого дня укажи 1–2 важных сигнала, если они есть в weekly JSON evidence.
- Если по дню нет evidence, напиши: `нет evidence в weekly JSON`.
- Не придумывай содержание дня вне weekly JSON.

---

## 3. Что реально происходило

Разбери неделю по блокам.

Используй только блоки, которые есть в weekly JSON.

Формат:

```markdown
### Работа / дорога / деньги

...

### Учёба / проекты

...

### Спорт / тело / энергия

...

### Отдых / игры / контент

...

### Социальное

...

### Быт / деньги

...
```

Если блок отсутствует в weekly JSON, не выдумывай. Можно написать:

```text
В weekly JSON нет отдельного блока по этой теме.
```

Но не надо заполнять пустой раздел длинным текстом.

---

## 4. Wins

Используй `main_wins`.

Не превращай обычный отдых, игры или контент в wins, если weekly JSON показывает tension вокруг этого.

Для каждого win:

```markdown
- Claim. Evidence: YYYY-MM-DD — "quote".
```

Если `main_wins` пустой:

```text
В weekly JSON не выделены отдельные wins.
```

Если `main_wins` НЕ пустой, обязательно перечисли каждый объект из `main_wins`.

Не пропускай wins из-за того, что они кажутся маленькими.
Не заменяй wins своим общим выводом.
Не пиши, что wins отсутствуют, если `main_wins` содержит элементы.

---

## 5. Главные напряжения

Используй `main_tensions`.

Дай 3–7 пунктов, если evidence позволяет.

Каждый пункт должен быть конкретным:

- скука;
- пустота;
- дорога;
- работа;
- сомнение;
- усталость;
- open loop;
- конфликт между действием и намерением.

Формат:

```markdown
- Скука и пустота повторялись в нескольких днях. Evidence: 2026-05-26, 2026-05-29.
```

Не драматизируй.

---

## 6. Эмоциональный фон и энергия

Используй:

- `emotional_background`;
- `body_energy`.

Раздели на два подпункта:

```markdown
### Эмоциональный фон

...

### Энергия / тело

...
```

Не делай медицинских или психологических выводов.

Не пиши “ежедневно”, если evidence не покрывает каждый день.

---

## 7. Фактический фокус недели

Используй `actual_focus`.

Ответь на вопрос:

```text
На что реально ушло время и внимание?
```

Не путай actual focus с желаниями, целями или красивой интерпретацией.

Если `actual_focus` содержит слишком общий claim, уточни его через evidence.

Если `actual_focus` пустой:

```text
В weekly JSON нет отдельного вывода по фактическому фокусу.
```

---

## 8. Повторяющиеся темы и противоречия

Используй:

- `repeated_topics`;
- `important_contradictions`.

Раздели:

```markdown
### Повторяющиеся темы

...

### Противоречия

...
```

Если `important_contradictions` пустой, не выдумывай.

Если contradiction есть, формулируй осторожно:

```text
Есть возможное противоречие между...
```

---

## 9. Open loops

Используй только поле `open_loops` из weekly JSON.

Open loop — это не просто эмоция и не общий вывод.  
Open loop — это конкретный незакрытый вопрос, решение, сомнение или тема, которая требует возвращения.

Один пункт = один open loop.

Если `open_loops` содержит элементы, ты обязан отразить каждый объект из `open_loops`.

Запрещено писать:

```text
Нет открытых вопросов или незавершённых действий, связанных с прошлой неделей.
```

если `open_loops.length > 0`.

Формат для непустого `open_loops`:

```markdown
- Вопрос / незакрытая тема. Evidence: YYYY-MM-DD — "quote".
```

Формулируй open loops конкретно.

Плохо:

```text
Отсутствие ясности в отношении смысла труда и личных целей.
```

если evidence говорит только про сомнение в работе.

Лучше:

```text
Остаётся вопрос, стоит ли текущая работа потраченных сил. Evidence: 2026-05-28 — "не знаю, стоит ли потраченное на работу денег сил."
```

Если `open_loops` пустой:

```text
В weekly JSON нет выделенных open loops.
```

Не придумывай open loops из `main_tensions`, `emotional_background` или `repeated_topics`, если они не вынесены в `open_loops`.

Однако если тебе кажется, что open loop должен был быть, но поле пустое, не добавляй его в отчёт. Вместо этого можно написать одну техническую строку:

```text
Возможные незакрытые вопросы лучше уточнить на этапе weekly aggregation.
```

Но не добавляй эту строку, если пользователь просит более чистый финальный отчёт без технических комментариев.

---

## 10. Разумный следующий шаг

Используй только `next_week_focus_candidates`.

Не придумывай новые рекомендации.

Если `next_week_focus_candidates` пустой, не придумывай рекомендации.

В таком случае напиши коротко:

```text
В weekly JSON нет выделенного следующего шага.
```

Не добавляй фразу про то, что это нужно исправить на этапе weekly aggregation.

Если поле не пустое, дай 1–3 пункта.

Каждый пункт должен быть маленьким и проверяемым через неделю.

---

## 11. Что не делать

Используй только `what_not_to_do`.

Не придумывай новые запреты.

Если `what_not_to_do` пустой:

```text
В weekly JSON нет отдельного списка what_not_to_do.
```

Если `what_not_to_do` НЕ пустой, обязательно отрази каждый объект из этого поля.

Если поле заполнено, формулируй как ограничение, а не как запрет на жизнь.

---

## 12. Evidence notes

Это обязательный раздел.

Нужно дать 7–14 evidence notes, если weekly JSON содержит достаточно evidence.

Цель: показать, на чём держится отчёт.

Правила:

- Покрой как можно больше дней из `days_included`.
- Если evidence есть по 7 дням, дай минимум 1 evidence note на каждый день.
- Если evidence есть только по 3 дням, дай notes только по ним и явно напиши, что evidence coverage ограничен.
- Не выбирай только 3 случайных цитаты, если weekly JSON содержит больше evidence.
- Не вставляй слишком длинные цитаты.
- Указывай source_field.

Формат:

```markdown
- 2026-05-25 — "quote" (`source_field`)
- 2026-05-26 — "quote" (`source_field`)
```

---

# 8. Запрещённые ошибки

Не делай так:

- не используй эмоджи;
- не используй декоративные иконки;
- не придумывай next steps, если `next_week_focus_candidates` пустой;
- не придумывай what not to do, если `what_not_to_do` пустой;
- не делай evidence notes только из 3 дней, если weekly JSON содержит evidence по большему числу дней;
- не делай claims шире evidence;
- не называй спорт “регулярным”, если evidence только за 1–2 дня;
- не пиши “ежедневно”, если evidence не покрывает каждый день;
- не называй отдых “минимальным”, если weekly JSON показывает игры / контент / социальную активность;
- не добавляй новые блоки, которых нет в weekly JSON;
- не превращай отчёт в пересказ каждого дня;
- не превращай отчёт в мотивационный текст.

---

# 9. Обязательный field audit перед написанием отчёта

Перед генерацией markdown мысленно проверь размеры ключевых массивов:

```text
main_wins.length = ...
main_tensions.length = ...
emotional_background.length = ...
body_energy.length = ...
actual_focus.length = ...
repeated_topics.length = ...
important_contradictions.length = ...
open_loops.length = ...
next_week_focus_candidates.length = ...
what_not_to_do.length = ...
```

Если length > 0, соответствующий раздел отчёта не может говорить, что данных нет.

Если length = 0, соответствующий раздел не должен придумывать содержание.
Особенно важно для `open_loops`:

- если `open_loops.length > 0`, раздел `## 9. Open loops` обязан содержать минимум один пункт;
- если `open_loops.length > 0`, запрещена фраза “нет открытых вопросов”;
- если `open_loops.length = 0`, не выдумывай open loops из tensions;
- не превращай open loop в абстрактную тему вроде “смысл жизни”, если evidence говорит о конкретной работе, деньгах, ремонте, учёбе или выборе.


---

# 10. Финальная проверка перед ответом

Перед тем как вернуть markdown, проверь:

1. В отчёте нет эмоджи.
2. Есть раздел `## 0. Покрытие evidence`.
3. Есть таблица `## 2. Карта недели по дням`.
4. В Evidence notes больше 3 пунктов, если weekly JSON содержит evidence больше чем по 3 дням.
5. Все дни из `days_included` представлены в карте недели.
6. Если `next_week_focus_candidates` пустой, ты не придумал рекомендации.
7. Если `what_not_to_do` пустой, ты не придумал запреты.
8. Нет claims шире evidence.
9. Нет медицинских или психологических диагнозов.
10. Если `main_wins` непустой, раздел Wins не говорит, что wins отсутствуют.
11. Если `what_not_to_do` непустой, раздел “Что не делать” его отражает.
12. Если `next_week_focus_candidates` пустой, рекомендации не придуманы.
13. Если `open_loops` непустой, раздел Open loops не говорит, что открытых вопросов нет.
14. Ответ — только markdown.
