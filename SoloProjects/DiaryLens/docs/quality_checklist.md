# DiaryLens Quality Checklist

## Purpose

Этот чеклист нужен для ручной оценки качества результата после каждого прогона DiaryLens MVP v0.1.

Он помогает одинаково проверять три уровня pipeline:

```text
daily JSON
weekly JSON
weekly_report.md
```

Чеклист не является README, инструкцией по запуску или описанием архитектуры. Это инструмент контроля качества outputs.

Главный вопрос чеклиста:

> Можно ли доверять результату weekly review, и если нет — на каком этапе pipeline возникла проблема?

---

## How to use

После команды:

```bash
diarylens run-week --week-id 2026-W22 --pdf data/raw/weekly/2026-W22.pdf
```

проверь результат в таком порядке:

1. `daily JSON`
2. `verified daily JSON`
3. `weekly JSON`
4. `weekly_report.md`

Рекомендуемый порядок оценки:

```text
daily JSON quality
→ weekly JSON quality
→ weekly_report.md quality
→ score
→ debug decision
```

Не начинай оценку с финального отчёта. Если daily JSON плохой, weekly JSON и report почти наверняка тоже будут искажены.

---

## Core quality principles

### 1. Evidence-first

Выводы должны опираться на цитаты, source-backed fragments или явно сохранённые элементы daily / weekly JSON.

Плохо:

```text
Неделя была тревожной.
```

Хорошо:

```text
В нескольких днях есть evidence про скуку, пустоту и сомнение в работе, поэтому можно осторожно говорить о напряжённом фоне.
```

### 2. No hallucinations

Нельзя добавлять факты, эмоции, причины, мотивы, выводы или советы, если их нет во входных данных.

### 3. Correct scope

Вывод не должен быть шире evidence.

- Evidence из 1 дня → локальный эпизод.
- Evidence из 2 дней → сигнал в нескольких днях.
- Evidence из 3+ дней → возможная weekly theme.
- Evidence из большинства дней → можно осторожно говорить о сути недели.

### 4. Practical usefulness

Отчёт должен помогать понять неделю и принять 1–3 разумных решения. Он не должен просто звучать красиво.

### 5. No motivational fluff

Запрещены мотивационная вода, generic self-help, коучинговые формулировки и абстрактные советы без evidence.

### 6. No over-analysis

DiaryLens не должен превращать дневник в диагноз, глубокий психологический разбор, анализ личности или попытку объяснить всё одной причиной.

### 7. Concrete next steps

Следующие шаги должны быть маленькими, конкретными, проверяемыми через неделю, основанными на weekly JSON и не шире evidence.

# 1. Daily JSON quality

Daily JSON отвечает на вопрос:

> Что реально было в этом дне?

На этом этапе не нужно делать большие недельные выводы. Главная задача — не потерять важные факты и сигналы дня.

---

## Coverage

Проверь:

- [ ] `important_moments` покрывают ядро дня.
- [ ] Значимые события не потеряны.
- [ ] Учёба / работа / проекты не потеряны, если были в дневнике.
- [ ] Деньги / быт / ремонт / покупки не потеряны, если были в дневнике.
- [ ] Здоровье / спорт / сон / усталость / восстановление не потеряны, если были.
- [ ] Социальные взаимодействия не потеряны, если были.
- [ ] Tensions / doubts / open loops не спрятаны только в `short_summary`.
- [ ] Сильные фразы из дневника попали в `key_quotes` или relevant fields.
- [ ] День не стал выглядеть продуктивнее, спокойнее или позитивнее, чем он был в исходном тексте.

---

## Classification

### Wins

- [ ] `wins` содержит только реальные wins.
- [ ] В `wins` не попали игры, отдых или контент, если они были связаны со скукой, избеганием или “не лучшее решение”.
- [ ] Маленькие реальные wins не потеряны: сделал рутину, сходил на тренировку, закрыл учебное дело, помог человеку.

### Tensions

- [ ] `tensions` содержит сомнения, напряжения, скуку, усталость, избегание, open loops.
- [ ] Страхи, фразы “не знаю”, “боюсь”, “не лучшее решение”, “стоит ли” не потеряны.
- [ ] Tensions не сглажены до нейтрального summary.

### Emotional signals

- [ ] `emotional_signals` / `emotions` не выдумывает эмоции.
- [ ] Эмоции взяты из текста или явно следуют из формулировки автора.
- [ ] Не добавлены диагнозы или глубокие психологические объяснения.

### Body / energy signals

- [ ] `body_energy_signals` содержит только сон, еду, спорт, усталость, здоровье, восстановление, самочувствие.
- [ ] Обычные действия вроде “поиграл”, “помылся”, “поел” не записаны как energy signal без явного сигнала тела / энергии.
- [ ] Не добавлены причины усталости, если они не указаны.

### Social signals

- [ ] `social_signals` содержит только реальных людей.
- [ ] AI / LLM / “иишка” не попадает в `social_signals`, если речь не о реальном человеке.
- [ ] Встречи, звонки, переписки, помощь близким не потеряны.

### Decisions

- [ ] `decisions` содержит только явные решения, планы или выборы.
- [ ] Не создаются decisions с пустой quote.
- [ ] Не записываются размышления как решения, если выбора не было.

### Open loops

- [ ] `open_loops` содержит только незакрытые вопросы, дела, сомнения или решения.
- [ ] Open loop сформулирован конкретно.
- [ ] Обычная эмоция не превращается в open loop.
- [ ] Вопросы про деньги, работу, ремонт, цели, учёбу не потеряны.

---

## Evidence

Проверь:

- [ ] Каждый item имеет `quote`.
- [ ] `quote` не пустой.
- [ ] `quote` основан на исходном тексте.
- [ ] `quote` достаточно короткий, но сохраняет смысл.
- [ ] `note` объясняет, почему quote попал в поле.
- [ ] `note` не добавляет новых фактов.
- [ ] `note` не превращает факт в психологическую интерпретацию.
- [ ] `short_summary` не содержит новых фактов вне items.

---

## Red flags

Красные флаги daily JSON:

- [ ] Пустой `tensions`, хотя в дне есть сомнения / скука / страх / “не лучшее решение”.
- [ ] Пустой `open_loops`, хотя есть незакрытый вопрос.
- [ ] `wins` содержит игры / отдых / контент как “успех”, хотя в тексте это было от скуки или избегания.
- [ ] `body_energy_signals` заполнен обычными действиями без телесного / энергетического сигнала.
- [ ] AI попал в `social_signals`.
- [ ] `decisions` содержит item с пустым quote.
- [ ] `body_energy_signals` содержит item с пустым quote.
- [ ] Summary звучит позитивнее или продуктивнее, чем день.
- [ ] Summary скрывает главное напряжение дня.
- [ ] Daily JSON выглядит как “красивый пересказ”, а не structured extraction.

---

# 2. Weekly JSON quality

Weekly JSON отвечает на вопрос:

> Что было главным содержанием недели?

Он должен агрегировать дни, а не писать финальный отчёт.

---

## Evidence-backed claims

Проверь:

- [ ] Каждый `WeeklyItem` имеет `evidence`.
- [ ] Evidence реально поддерживает `summary`.
- [ ] `source_day_md` не `null`.
- [ ] `source_daily_json` не `null`.
- [ ] `source_field` указывает на актуальное daily-поле.
- [ ] Нет claims без evidence.
- [ ] Evidence не используется как украшение.
- [ ] `quote` в evidence не пустой.
- [ ] `note` не добавляет новых фактов.
- [ ] Один weekly item не склеивает несколько разных claims, если evidence покрывает только один.

---

## Scope

Проверь:

- [ ] Evidence из 1 дня не обобщается на всю неделю.
- [ ] Evidence из 2 дней формулируется как repeated signal in several days.
- [ ] Evidence из 3+ дней может быть weekly theme.
- [ ] `week_essence` не строится на 1–2 днях.
- [ ] `actual_focus` не строится на случайных ярких цитатах.
- [ ] В summary нет слов “преимущественно”, “вся неделя”, “главный фокус”, если evidence слабая.
- [ ] Missing days учитываются осторожно.
- [ ] Локальные эпизоды не превращаются в weekly pattern.

---

## Field quality

### `week_essence`

- [ ] Не звучит философски.
- [ ] Отражает фактические блоки недели.
- [ ] Не строится на 1–2 днях.
- [ ] Не звучит как финальный report.

### `main_events`

- [ ] Отражает фактические блоки недели.
- [ ] Работа / дорога отражены, если занимали заметное место.
- [ ] Учёба / проекты отражены, если были.
- [ ] Спорт / тело / энергия отражены, если были.
- [ ] Социальный блок отражён, если был в 2+ днях.
- [ ] Быт / деньги / ремонт отражены, если были значимыми.

### `main_wins`

- [ ] Не включает игры / отдых / контент, если они были связаны со скукой или избеганием.
- [ ] Не теряет маленькие реальные wins.
- [ ] Не превращает “было приятно” в win без действия / результата.
- [ ] Не превращает обычную занятость в прогресс.

### `main_tensions`

- [ ] Не теряет скуку.
- [ ] Не теряет пустоту.
- [ ] Не теряет усталость.
- [ ] Не теряет сопротивление.
- [ ] Не теряет сомнения.
- [ ] Не теряет open loops.
- [ ] Не сглаживает неприятные сигналы ради позитивного отчёта.

### `emotional_background`

- [ ] Эмоции сгруппированы по кластерам, если их несколько.
- [ ] Не смешивает разные состояния в один расплывчатый claim.
- [ ] Не добавляет диагнозов.
- [ ] Не делает глубоких психологических объяснений.

### `body_energy`

- [ ] Содержит только сон, спорт, еду, усталость, здоровье, восстановление, тело.
- [ ] Не добавляет причинность без evidence.
- [ ] Не называет активность регулярной, если evidence только из 1–2 дней.

### `study_and_projects`

- [ ] Не пустой, если в daily JSON были study / ML / DS / project signals.
- [ ] Единичный учебный эпизод сформулирован как точечный, а не как главный фокус недели.
- [ ] Разговор с AI не превращён в социальный сигнал.

### `social_context`

- [ ] Содержит только реальные взаимодействия с людьми.
- [ ] Не включает AI как social.
- [ ] Не теряет друзей, семью, одногруппников, коллег, помощь.

### `actual_focus`

- [ ] Отвечает на вопрос “на что реально ушло время”.
- [ ] Не звучит как “личное развитие”, если evidence про работу / дорогу / игры.
- [ ] Отделяет фактический фокус от желаемого фокуса.
- [ ] Строится по day-by-day map, а не по одной красивой цитате.

### `repeated_topics`

- [ ] Содержит только темы, которые повторялись.
- [ ] Тема из одного дня не попала как repeated topic.
- [ ] Игры / контент отражены, если повторялись.
- [ ] Скука / пустота отражены, если повторялись.
- [ ] Работа / дорога / деньги отражены, если повторялись.

### `important_contradictions`

- [ ] Основаны на evidence, а не на догадках.
- [ ] Не пустые автоматически, если есть явный conflict / avoidance / “не лучшее решение”.
- [ ] Формулируются осторожно.
- [ ] Не превращаются в психологический диагноз.

### `open_loops`

- [ ] Один open loop = один незакрытый вопрос.
- [ ] Не склеивает разные вопросы в один.
- [ ] Не превращает эмоцию в open loop.
- [ ] Не теряет вопросы про работу, деньги, ремонт, учёбу, цели.
- [ ] Сформулирован конкретно.

### `risks_next_week`

- [ ] Риск следует из evidence.
- [ ] Риск не звучит драматично.
- [ ] Риск не превращается в совет.

### `next_week_focus_candidates`

- [ ] Конкретны и проверяемы.
- [ ] Основаны на evidence.
- [ ] Не звучат как generic advice.
- [ ] Не больше 1–3 пунктов.
- [ ] Не содержат “стать продуктивнее”, “больше отдыхать”, “развиваться”.

### `what_not_to_do`

- [ ] Находится в зоне контроля пользователя.
- [ ] Не звучит как запрет на жизнь.
- [ ] Не запрещает работу / учёбу / людей / отдых целиком.
- [ ] Основано на evidence.

---

## Red flags

Красные флаги weekly JSON:

- [ ] `week_essence` звучит философски.
- [ ] `week_essence` построен на 1–2 днях.
- [ ] `actual_focus` звучит как “личное развитие”, но evidence про работу / дорогу / игры.
- [ ] `actual_focus` не отвечает на вопрос “на что ушло время”.
- [ ] `repeated_topics` содержит тему из одного дня.
- [ ] `main_wins` включает игры / контент / отдых как progress без evidence.
- [ ] `main_tensions` пустой или слишком мягкий, хотя daily JSON содержит напряжения.
- [ ] `important_contradictions` пустой, хотя есть явный conflict / avoidance / “не лучшее решение”.
- [ ] `open_loops` склеивает разные вопросы в один.
- [ ] `next_week_focus_candidates` звучат как generic advice.
- [ ] `what_not_to_do` звучит как запрет на жизнь.
- [ ] `source_daily_json` равен `null`.
- [ ] Evidence покрывает только 1–3 дня, но выводы звучат как про всю неделю.

# 3. Weekly report quality

Weekly report отвечает на вопрос:

> Что я должен понять про эту неделю после чтения отчёта?

Он должен быть понятным, конкретным и полезным для weekly review.

---

## Structure

Проверь:

- [ ] Есть суть недели.
- [ ] Есть карта / фактическое описание недели.
- [ ] Есть wins.
- [ ] Есть tensions.
- [ ] Есть emotional / body-energy section.
- [ ] Есть actual focus.
- [ ] Есть repeated topics / contradictions.
- [ ] Есть open loops.
- [ ] Есть next step или честно указано, что в weekly JSON он не выделен.
- [ ] Есть what not to do или честно указано, что в weekly JSON его нет.
- [ ] Есть evidence notes.
- [ ] Нет эмоджи и декоративных иконок.
- [ ] Markdown читаемый и не перегружен.

---

## Content quality

Проверь:

- [ ] Report не добавляет факты вне weekly JSON.
- [ ] Report не делает выводы шире evidence.
- [ ] Report конкретный, а не шаблонный.
- [ ] Report не пересказывает каждый день полностью.
- [ ] Report показывает, что реально происходило.
- [ ] Report отделяет фактический фокус от желаемого фокуса.
- [ ] Report сохраняет важные tensions.
- [ ] Report не превращается в мотивационный текст.
- [ ] Report не скрывает negative / tension signals.
- [ ] Report не звучит как generic self-help.
- [ ] Report не делает диагнозов.
- [ ] Report не даёт советы, которых нет в weekly JSON.
- [ ] Если `main_wins` непустой, report не пишет, что wins отсутствуют.
- [ ] Если `open_loops` непустой, report не пишет, что open loops отсутствуют.
- [ ] Если `what_not_to_do` непустой, report не пишет, что what not to do отсутствует.

---

## Evidence quality

Проверь:

- [ ] Evidence notes есть.
- [ ] Evidence notes покрывают большинство дней, если evidence есть по большинству дней.
- [ ] Evidence notes не ограничены случайными 3 цитатами.
- [ ] Цитаты короткие и понятные.
- [ ] Evidence notes связаны с главными claims.
- [ ] В отчёте видно, какие даты поддерживают важные выводы.
- [ ] Один день не превращён в вывод о всей неделе.

---

## Next steps quality

Проверь:

- [ ] 1–3 шага максимум.
- [ ] Каждый шаг конкретный.
- [ ] Каждый шаг можно проверить через неделю.
- [ ] Каждый шаг основан на weekly JSON.
- [ ] Нет советов вроде “стать продуктивнее”, “больше отдыхать”, “развиваться”.
- [ ] Если `next_week_focus_candidates` пустой, report не придумывает рекомендации.

---

## Open loops quality

Проверь:

- [ ] Open loops берутся из `weekly_json.open_loops`.
- [ ] Один пункт = один незакрытый вопрос.
- [ ] Open loop сформулирован конкретно.
- [ ] Open loop не звучит как общая философская тема.
- [ ] Если open loops пустые, report не выдумывает их из tensions.
- [ ] Если open loops непустые, report не пишет, что открытых вопросов нет.

---

## Red flags

Красные флаги weekly report:

- [ ] Отчёт звучит как generic self-help.
- [ ] Слишком много абстрактных слов.
- [ ] Есть эмоджи или декоративные иконки.
- [ ] Нет evidence notes.
- [ ] Evidence notes только из 1–3 дней, хотя weekly JSON содержит больше evidence.
- [ ] Claims не связаны с evidence.
- [ ] Один день превращён в вывод о всей неделе.
- [ ] Report игнорирует negative / tension signals.
- [ ] Report даёт советы, которых нет в weekly JSON.
- [ ] Report пишет, что wins / open loops / what not to do отсутствуют, хотя в weekly JSON они есть.
- [ ] Report звучит красиво, но после чтения непонятно, что реально происходило.

---

# 4. Simple scoring

Оцени отдельно каждый уровень:

```text
Daily JSON score:
Weekly JSON score:
Report score:
```

Используй шкалу:

## A — можно использовать без правок

- evidence покрывает основные события;
- нет hallucinations;
- scope корректный;
- важные tensions не потеряны;
- report полезен для weekly review;
- правки не требуются или минимальны.

## B — годно, но есть мелкие неточности

- в целом можно использовать;
- есть 1–3 небольшие ошибки;
- отдельные формулировки можно поправить;
- нет серьёзных hallucinations;
- отчёт не вводит в заблуждение.

## C — полезно частично, нужно править prompt или weekly JSON

- часть полезных сигналов есть;
- потеряны важные события или tensions;
- есть claims шире evidence;
- report местами звучит generic;
- нужны prompt changes.

## D — отчёт вводит в заблуждение

- есть hallucinations;
- важные события потеряны;
- выводы противоречат evidence;
- report даёт советы без weekly JSON;
- один день превращён в вывод о всей неделе;
- нельзя использовать для weekly review без серьёзной правки.

---

## Overall decision

После оценки выбери один общий вывод:

```text
MVP pass
needs prompt tuning
needs schema change
needs extraction fix
needs splitter/cleaning fix
```

### MVP pass

Используй, если:

- Daily JSON score A/B;
- Weekly JSON score A/B;
- Report score A/B;
- нет систематических hallucinations.

### needs prompt tuning

Используй, если структура данных нормальная, schema подходит, но LLM неправильно формулирует или агрегирует.

### needs schema change

Используй, если нужные данные негде хранить, поля путаются, prompt постоянно вынужден делать awkward mapping или Pydantic validation мешает выразить нужный output.

### needs extraction fix

Используй, если daily JSON теряет события, daily JSON неправильно классифицирует fields или daily verification не исправляет obvious mistakes.

### needs splitter/cleaning fix

Используй, если дни выделены неправильно, записи смешались, текст грязный или пропали фрагменты исходного дневника.

---

# 5. Debug decision tree

Используй это дерево, чтобы понять, где проблема.

```text
Report плохой?
│
├─ Weekly JSON хороший?
│  ├─ Да → править weekly_report prompt.
│  └─ Нет → смотреть Weekly JSON.
│
├─ Weekly JSON бедный?
│  ├─ Да → править weekly_aggregation prompt.
│  └─ Нет → смотреть report prompt.
│
├─ Weekly JSON потерял события?
│  ├─ Да → проверить daily JSON.
│  └─ Нет → проверить claim scope / evidence rules.
│
├─ Daily JSON потерял события?
│  ├─ Да → править daily_extraction / daily_verification prompt.
│  └─ Нет → проверить weekly aggregation.
│
├─ Daily JSON неправильно классифицирует поля?
│  ├─ Да → править daily_extraction prompt или schema mapping.
│  └─ Нет → проверить weekly prompt.
│
├─ Дни неверно выделены?
│  ├─ Да → править day_splitter.
│  └─ Нет → проверить extraction.
│
└─ Текст грязный?
   ├─ Да → править pdf extraction / cleaning.
   └─ Нет → проблема, скорее всего, в prompt quality gates.
```

# 6. MVP acceptance criteria

MVP v0.1 можно считать стабильным, если:

- [ ] `run-week` работает на 2–3 разных неделях.
- [ ] Pipeline стабильно создаёт `data/reports/{week_id}_weekly_report.md`.
- [ ] Daily JSON обычно получает A/B.
- [ ] Weekly JSON обычно получает A/B.
- [ ] Weekly report обычно получает A/B.
- [ ] Нет систематических hallucinations.
- [ ] Evidence покрывает большинство дней.
- [ ] Report не добавляет факты вне weekly JSON.
- [ ] Report не даёт советы вне weekly JSON.
- [ ] Report не звучит как generic self-help.
- [ ] Report реально полезен для weekly review.
- [ ] Ошибки, если появляются, понятно локализуются через debug decision tree.
