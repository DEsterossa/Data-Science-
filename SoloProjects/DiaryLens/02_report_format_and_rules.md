# DiaryLens — формат ответов, отчётов и правила качества

## 1. Источник истины

Главный источник истины после обработки PDF:

```text
data/processed/days_md/YYYY-MM-DD.md
```

То есть исходный дневной markdown важнее любого сжатого JSON.

Daily JSON используется как навигационный слой:

```text
data/processed/days_json/YYYY-MM-DD.json
```

Он помогает понять:

- какие важные события были в дне;
- какие темы затронуты;
- какие были tension / wins / open questions;
- какой день стоит открыть для ответа.

Weekly JSON используется только как optional weekly snapshot:

```text
data/processed/weeks_json/{week_id}.json
```

Он не должен быть единственным источником для `ask_history`, потому что сжимает данные и может терять важные детали.

Главный принцип:

```text
raw day markdown → answer with evidence
```

Не:

```text
weekly JSON → answer as if это вся реальность
```

---

## 2. Главный формат будущего продукта: ask_history

Новый главный режим проекта:

```text
question
→ retrieve relevant days / chunks
→ answer from raw day markdown
→ evidence with dates and quotes
```

Рекомендуемый формат ответа:

```markdown
# Answer

Короткий прямой ответ на вопрос.

## Evidence

- YYYY-MM-DD — “короткая цитата” (`source_day_md`)
- YYYY-MM-DD — “короткая цитата” (`source_day_md`)

## Interpretation

Осторожный вывод на основе найденных фрагментов.

## Limitations

Что не найдено / где evidence слабая / какой период покрыт не полностью.
```

Важно: если retrieval нашёл мало данных, система должна прямо сказать, что вывод предварительный.

---

## 3. Правило evidence

Каждый важный вывод должен иметь опору.

Плохо:

```text
Ты боялся будущего.
```

Лучше:

```text
В найденных записях от 2026-06-10 и 2026-06-13 несколько раз появляется тема неопределённости с проектом и выбором ML-направления. Поэтому можно осторожно сказать, что вопрос фокуса был заметным напряжением этого периода.
```

То есть:

```text
вывод → evidence → осторожная интерпретация
```

Не “я знаю, что с тобой”.

А:

```text
по найденным записям видно вот это
```

---

## 4. Scope rule

Вывод не должен быть шире evidence.

Правила:

- evidence из 1 дня → локальный эпизод;
- evidence из 2 дней → repeated signal в нескольких днях;
- evidence из 3+ дней → заметная тема периода;
- evidence из 4+ дней или большинства найденных дней → можно говорить о period-level pattern.

Плохо:

```text
Ты всегда уходишь в скуку после ярких событий.
```

если evidence только за один день.

Лучше:

```text
В одном найденном дне скука появилась после насыщенного периода. Этого недостаточно, чтобы считать это устойчивым паттерном.
```

## 5. Weekly report как optional snapshot

Weekly report остаётся полезным, но больше не является главной функцией проекта.

Актуальный weekly report для v0.1 может сохраняться в:

```text
data/reports/{week_id}_weekly_report.md
```

Роль weekly report:

```text
быстро зафиксировать неделю
дать обзор
создать readable snapshot
```

Ограничение:

> Если weekly report генерируется только из weekly JSON, он может быть сухим и терять важные детали.

Поэтому для более качественного отчёта допустим гибридный режим:

```text
weekly JSON
+ selected daily JSONs
+ selected raw day markdown fragments
→ weekly_report.md
```

При этом report не должен превращаться в свободный психологический разбор без evidence.

---

## 6. Рекомендуемая структура weekly snapshot

Если weekly report всё ещё генерируется, структура может быть такой:

```markdown
# Weekly Snapshot — {week_id}

## 0. Coverage

- included days
- missing days
- evidence coverage

## 1. Суть недели

Короткий вывод, не шире evidence.

## 2. Карта дней

День → важные события / темы → ключевой evidence.

## 3. Что реально происходило

Фактические блоки недели.

## 4. Wins / progress

Только то, что подтверждено evidence.

## 5. Tensions / open loops

Что давило, раздражало, утомляло или осталось незакрытым.

## 6. Фактический фокус

На что реально ушло внимание.

## 7. Повторяющиеся темы

Только если есть evidence минимум из нескольких дней.

## 8. Следующий шаг

1–3 кандидата, если они реально поддержаны evidence.

## 9. What not to do

Только если это поддержано evidence.

## 10. Evidence notes

Короткие цитаты с датами.
```

---

## 7. Что система НЕ должна делать

Система не должна:

- ставить диагнозы;
- притворяться психологом;
- делать категоричные выводы без evidence;
- превращать отчёт или ответ в мотивационную воду;
- каждый раз советовать “просто отдохнуть”;
- раздувать список задач;
- предлагать 10 направлений;
- усиливать тревогу;
- делать вид, что любая активность — это прогресс;
- анализировать ради анализа;
- добавлять факты, которых нет в найденных источниках;
- отвечать по summary, если доступен raw day markdown;
- делать выводы по weekly JSON так, будто это полный дневник.

Главное ограничение:

> система должна помогать принимать решения и находить evidence, а не создавать ещё один слой бесконечного самоанализа.

## 8. Quality checklist для ask_history

Качество `ask_history` проверяется по вопросам:

| Метрика | Вопрос |
|---|---|
| retrieval_relevance | Найденные дни / chunks реально относятся к вопросу? |
| evidence_quality | Есть ли цитаты и даты, подтверждающие ответ? |
| source_fidelity | Ответ не добавляет факты вне найденных источников? |
| scope_control | Вывод не шире evidence? |
| usefulness | Ответ помогает понять прошлые данные? |
| limitations | Система честно пишет, если данных мало? |

Простая шкала:

```text
A — ответ можно использовать без правок
B — полезно, но есть мелкие неточности
C — частично полезно, retrieval / prompt надо улучшать
D — ответ вводит в заблуждение
```

---

## 9. Quality checklist для daily JSON

Daily JSON должен оцениваться не как “идеальный пересказ дня”, а как day index.

Проверять:

- важные события дня не потеряны;
- темы дня отражены достаточно явно;
- open questions сохранены;
- key quotes полезны для будущего поиска;
- нет слишком смелых интерпретаций;
- `source_day_md` указан корректно.

Daily JSON может быть неполным как отчёт, но должен быть полезным как навигационная карта дня.

---

## 10. Quality checklist для weekly JSON / weekly report

Weekly JSON / report теперь имеют более низкий статус.

Оценивать:

- не вводит ли weekly snapshot в заблуждение;
- не теряет ли важные темы;
- не делает ли выводы шире evidence;
- не выглядит ли report слишком сухим из-за пересжатия;
- нужен ли raw-fragment hybrid mode для улучшения качества.

---

## 11. Правила тона

Тон должен быть:

- спокойным;
- доказательным;
- прямым, но не агрессивным;
- практичным;
- без мистификации;
- без “психологического всезнания”;
- без пафосных мотивационных формулировок.

Хороший ответ не должен звучать как мотивационный пост.

Он должен звучать как честная аналитическая сводка по найденным дневниковым данным.

---

## 12. Правило для CursorAI

При генерации кода через CursorAI важно не просить:

```text
сделай идеальную систему анализа дневника
```

Вместо этого задачи должны быть маленькими и проверяемыми.

Каждая задача должна иметь:

- context;
- task;
- files;
- constraints;
- done criteria;
- тесты или способ проверки.

Примеры:

```text
Сделай модуль, который читает days_md и создаёт raw_day_chunk документы для retrieval.
```

```text
Сделай CLI-команду search-memory, которая ищет top-k chunks по cosine similarity и выводит date, score, source_day_md, text preview.
```
