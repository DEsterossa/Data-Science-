## 1. Поиск токсичных комментариев (Yandex Practicum)

### 1. Задача проекта
- **Проблема**: нужно автоматически находить токсичные комментарии пользователей (оскорбления, хейт) и отправлять их на модерацию.
- **Почему важно**: поддержание качества контента и пользовательского опыта, снижение нагрузки на модераторов.

### 2. Данные
- **Источник**: файл `toxic_comments.csv` из учебного сервиса "Викишоп".
- **Тип**: текст комментария (`text`) + бинарная разметка токсичности (`toxic` -- 0 или 1).
- **Размер**: крупный датасет (десятки тысяч комментариев).
- **Особенности**:
  - дисбаланс классов (токсичных меньше);
  - живой язык: опечатки, сленг, эмодзи;
  - разная длина текстов.

### 3. Подготовка данных
- **Preprocessing**:
  - базовая очистка текста (lowercase, удаление лишних символов);
  - разбиение на train/valid/test;
  - учёт дисбаланса через выбор метрики и параметры моделей (`class_weight`).
- **Фичи**:
  - TF-IDF представление текста (`TfidfVectorizer`);
  - BERT-эмбеддинги (через Hugging Face) для части экспериментов.

### 4. Модель
- **Модели с TF-IDF**:
  - `LogisticRegression` (финальная модель),
  - `LinearSVC`,
  - `SGDClassifier`,
  - `MultinomialNB`.
- **Модели с BERT**:
  - fine-tuning трансформера через `transformers` + `Trainer` (с метрикой F1),
  - `LogisticRegression` по эмбеддингам BERT.
- **Как работает финальная модель**:
  - TF-IDF превращает текст в разреженный вектор важностей слов;
  - логистическая регрессия с L1-регуляризацией учит линейную границу токсично/нет, обнуляя лишние веса.
- **Библиотеки**: `pandas`, `numpy`, `sklearn`, `matplotlib/plotly`, `transformers`, `torch`, `accelerate`.

### 5. Метрики
- **Основная метрика**: F1-score (цель -- F1 >= 0.75).
- **Почему**:
  - классы несбалансированы, accuracy не информативна;
  - F1 балансирует precision/recall, что критично: и пропуск токсичных, и ложная модерация -- плохие исходы.

### 6. Результат
- **Качество**:
  - `LogisticRegression` + TF-IDF: F1 ~ **0.78** (валидация), F1 ~ **0.774** (тест);
  - fine-tuned BERT: F1 до **0.81**, но сильно дороже по ресурсам;
  - другие классические модели -- хуже по F1.
- **Интерпретация**:
  - модель уверенно превышает бизнес-порог (0.75), остаётся быстрой и простой для продакшена.

### 7. Почему выбрана эта модель
- Хороший **баланс качество/сложность/скорость**:
  - F1 ~0.78 при очень быстром обучении и инференсе;
  - не требует GPU и тяжёлого деплоя, в отличие от полного fine-tuning BERT;
  - L1-регуляризация даёт интерпретируемые веса по словам.
- В контексте учебного/прикладного сервиса это практичнее, чем тяжёлый BERT-пайплайн.

### 8. Сложности
- Работа с **дисбалансом классов** и правильный выбор метрики.
- Настройка разных пайплайнов (TF-IDF vs BERT) и требований к ресурсам.
- Риск **переобучения** при fine-tuning BERT на ограниченном числе токсичных примеров.

### 9. Улучшения
- Попробовать более сильные русскоязычные модели (ruBERT, ruRoberta, mDeBERTa) с аккуратным fine-tuning-ом и k-fold CV.
- Улучшить препроцессинг: нормализация сленга/эмодзи, орфокоррекция.
- Ввести калибровку вероятностей и оптимизацию порога под конкретный trade-off.
- Настроить MLOps: мониторинг F1 и retraining при дрейфе данных.

### 10. Краткий питч (30-40 секунд)
"Проект по поиску токсичных комментариев для сервиса наподобие "Викишопа". Взял размеченный датасет комментариев, сделал базовый препроцессинг и строил TF-IDF-признаки, параллельно экспериментируя с BERT. Сравнивал несколько моделей, основная метрика -- F1 из-за дисбаланса классов. Лучшей по балансу качества и ресурсов оказалась логистическая регрессия с L1-регуляризацией на TF-IDF: F1 около 0.78 на тесте при очень быстром обучении и предсказании. BERT давал немного выше F1, но стоимость и сложность пайплайна для реального сервиса оказались выше, поэтому финально выбрал классическую модель."

## 2. Персональный RAG-ассистент в Telegram (`SoloProjects/ga_rag_bot`)

### 1. Задача проекта
- **Проблема**: нужен ассистент, который отвечает на вопросы строго по пользовательским документам, а не "галлюцинирует".
- **Почему важно**: корпоративные базы знаний, юридические/технические документы -- критична привязка к источнику, а не выдуманные ответы.

### 2. Данные
- **Источник**: документы, которые пользователь загружает боту (`.txt`, `.pdf`, `.docx`).
- **Тип**: неструктурированный текст.
- **Размер**: произвольный, зависит от пользователя (в PoC -- документ о компании "Космокофе").
- **Особенности**:
  - разные форматы и структура;
  - возможные артефакты при конвертации PDF/DOCX в текст;
  - данные изолированы по пользователям (важна мультипользовательская архитектура).

### 3. Подготовка данных
- **Загрузка и парсинг**:
  - выбор loader-а по расширению (`TextLoader`, `PyPDFLoader`, `Docx2txtLoader`);
  - чтение документа в список `Document`.
- **Разбиение на чанки**:
  - `RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)`;
  - добавление метаданных (`doc_id`, `file_name`) ко всем чанкам.
- **Эмбеддинги и векторное хранилище**:
  - эмбеддинги с `HuggingFaceEmbeddings` (`cointegrated/rubert-tiny2`, нормализованные векторы);
  - хранение в FAISS с возможностью дозаписи: при повторной загрузке документа индекс дополняется новыми чанками.

### 4. Модель
- **Архитектура**: Retrieval-Augmented Generation (RAG).
- **Компоненты**:
  - Retriever: `FAISS.as_retriever(k=3)`;
  - LLM: `ChatGoogleGenerativeAI` (Gemini 2.5 Flash, `temperature=0.1`);
  - Prompt: `ChatPromptTemplate` с инструкцией использовать только контекст и не придумывать ответы.
- **Цепочка**:
  - `{"context": retriever | format_docs, "question": RunnablePassthrough()} | prompt | llm | StrOutputParser()`.
- **Интерфейс**:
  - Telegram-бот через `python-telegram-bot` (`ApplicationBuilder`, `CommandHandler`, `MessageHandler`);
  - отдельные хранилища эмбеддингов на пользователя.

### 5. Метрики
- Формальная автоматическая метрика в PoC не реализована.
- Оценка качества:
  - релевантность ответа вопросу;
  - отсутствие галлюцинаций;
  - корректное поведение при отсутствии ответа в тексте.
- Для продакшена подходят: Precision@k/Recall@k для retriever-а, human-rated quality для генератора.

### 6. Результат
- Бот:
  - корректно обрабатывает `.txt/.pdf/.docx`;
  - строит и обновляет векторные индексы по пользователям;
  - даёт ответы строго по документу;
  - честно говорит, что не знает, если ответа нет в тексте.
- В демо-кейсе по "Космокофе" правильно отвечает о миссии компании, ценах, ограничениях доставки.

### 7. Почему выбрана эта архитектура
- RAG:
  - решает проблему галлюцинаций LLM за счёт явного retrieval-а;
  - позволяет легко обновлять знания -- достаточно пересобрать индекс.
- Выбор `rubert-tiny2`:
  - хороший компромисс для русского языка между качеством и скоростью.
- FAISS:
  - стандарт для быстрых векторных поисков локально.
- LangChain + Telegram-бот:
  - упрощают оркестрацию и интеграцию в пользовательский интерфейс.

### 8. Сложности
- Поддержка разных форматов файлов и борьба с артефактами при парсинге.
- Персистентность и версионирование индексов: дозапись, обработка повреждённых индексов.
- Настройка промпта, чтобы LLM реально опирался на контекст, а не отвечал "из головы".

### 9. Улучшения
- Добавить переранжировщик (cross-encoder) поверх retriever-а.
- Ввести автоматический evaluation-контур на собранных парах вопрос-ответ-документ.
- Реализовать фильтрацию по `doc_id`, чтобы управлять несколькими документами пользователя.
- Добавить режимы объяснения и отсылки к конкретным фрагментам текста (citation).

### 10. Краткий питч (30-40 секунд)
"Это Telegram-бот в архитектуре RAG: пользователь загружает документ, я разбиваю его на чанки, считаю эмбеддинги `rubert-tiny2` и сохраняю их в FAISS-индекс для конкретного пользователя. При вопросе бот делает семантический поиск по этим чанкам, подставляет топ-3 в промпт и вызывает Gemini 2.5 Flash с инструкцией отвечать только по контексту. В результате бот умеет отвечать строго по пользовательским документам и честно говорит, если ответа в тексте нет."

## 3. Бинарная сегментация кожных образований PH2 (`SoloProjects/ph2`)

### 1. Задача проекта
- **Проблема**: сегментация кожных образований на дерматоскопических изображениях (lesion vs background).
- **Почему важно**: точная сегментация -- основа для автоматического анализа меланомы, измерения площади, формы и других клинических признаков.

### 2. Данные
- **Источник**: открытый медицинский датасет **PH2**.
- **Тип**:
  - цветные дерматоскопические изображения;
  - бинарные маски lesion;
  - дополнительные маски по цветовым классам (для информации).
- **Размер**: несколько сотен изображений (малый датасет для DL).
- **Особенности**:
  - небольшой объём данных -- высокий риск переобучения;
  - нестандартные разрешения;
  - важность точных границ маски.

### 3. Подготовка данных
- **Dataset и загрузка**:
  - класс `PH2SegDataset`, который ищет пары `*_Dermoscopic_Image` / `*_lesion` в нужной структуре папок;
  - возвращает `(image, mask)` в виде тензоров.
- **Препроцессинг**:
  - `resize_with_pad`: ресайз с сохранением пропорций и паддингом фона;
  - разные режимы интерполяции: BILINEAR для изображений, NEAREST для масок;
  - приведение маски к бинарному формату {0,1}, форма `1xHxW`.
- **Dataloaders**:
  - train/val-сплит по `val_ratio` с фиксированным seed;
  - `DataLoader` с нужными `batch_size`, `num_workers`, `pin_memory`.

### 4. Модель
- **Baseline-архитектура**:
  - компактный `UNetSmall`, обучаемый с нуля;
  - параметр `base_ch` регулирует ширину (capacity).
- **Продвинутая модель**:
  - U-Net из `segmentation_models_pytorch` с предобученным encoder-ом ResNet34 (ImageNet);
  - decoder и голова обучаются на PH2.
- **Loss и метрики**:
  - loss: BCE, Dice, комбинированный BCE+Dice;
  - метрики: soft/hard Dice, soft/hard IoU.
- **Инструменты**: PyTorch, `segmentation_models_pytorch`, `torchvision`, NumPy, Matplotlib.

### 5. Метрики
- **Основные**: Dice и IoU.
- **Почему**:
  - напрямую измеряют перекрытие предсказанной и истинной масок;
  - устойчивы к дисбалансу "фон vs объект".

### 6. Результат
- **Baseline U-Net (с нуля)**:
  - hard Dice ~ **0.88**;
  - hard IoU ~ **0.79**.
- **U-Net + ResNet34 encoder**:
  - hard Dice ~ **0.95**;
  - hard IoU ~ **0.90**.
- **Дополнительные выводы**:
  - BCE+Dice даёт лучший hard Dice (~0.84) для baseline;
  - увеличение ширины (`base_ch=16,32,64`) почти не даёт прироста, но может вести к переобучению;
  - увеличение порога бинаризации до ~0.7 немного улучшает hard-метрики.

### 7. Почему выбрана эта модель
- U-Net -- стандарт для медицинской сегментации: хорошо работает на небольших датасетах.
- На таком объёме данных ключевой фактор качества -- **предобученный encoder**, а не просто увеличение capacity:
  - ResNet34, обученный на ImageNet, даёт гораздо более информативные признаки;
  - это даёт резкий прирост Dice/IoU по сравнению с моделью с нуля.

### 8. Сложности
- Работа с малым датасетом и борьба с переобучением.
- Корректный ресайз изображений и масок, чтобы не ломать форму и границы lesion.
- Согласование метрик с визуальным качеством: подбор loss-ов и порога бинаризации.

### 9. Улучшения
- Добавить аугментации (flip, rotation, color jitter, elastic transforms).
- Потестировать более сложные архитектуры (UNet++, DeepLabv3+, FPN).
- Использовать post-processing масок (морфологические операции, CRF).
- Применить k-fold cross-validation для более надёжной оценки.

### 10. Краткий питч (30-40 секунд)
"Это playground по бинарной сегментации кожных образований на датасете PH2. Я собрал PyTorch-пайплайн с кастомным `PH2SegDataset`, аккуратным ресайзом и train/val-сплитом. В качестве baseline обучил компактный U-Net с нуля и получил Dice около 0.88, затем использовал U-Net с предобученным ResNet34 в качестве encoder-а и поднял Dice до ~0.95 и IoU до ~0.9. Также сравнивал разные loss-функции, capacity и пороги бинаризации; главный вывод -- на маленьком медицинском датасете качество решается использованием предобученного encoder-а, а не просто увеличением размера модели."

## 4. RoadGuard AI: Детектор объектов на дороге (`SoloProjects/road_object_detector`)

### 1. Задача проекта
- **Проблема**: нужно **локализовать** (обнаружить и обвести рамкой) множество объектов на дорожных сценах: автомобили, пешеходов, грузовики, велосипедистов и светофоры.
- **Почему важно**: такие системы лежат в основе ADAS (ассистентов водителя), анализа трафика и автопилотирования.

### 2. Данные
- **Источник**: датасет дорожных сцен с аннотациями в формате bounding box (CSV: `filename`, `width`, `height`, `class`, `xmin`, `ymin`, `xmax`, `ymax`).
- **Тип**: цветные фотографии дорожных сцен (512x512) + координаты bbox-ов по каждому объекту.
- **Классы**: `car`, `pedestrian`, `truck`, `biker`, `traffic_light` (5 классов после объединения).
- **Особенности**:
  - сильный **классовый дисбаланс** (машин намного больше, чем байкеров или грузовиков);
  - изначально избыточная гранулярность классов светофоров -- объединены в один класс `traffic_light`;
  - на одном изображении может быть множество объектов разных классов.

### 3. Подготовка данных
- **Очистка и предобработка аннотаций**:
  - объединение мелких классов светофоров в общий `traffic_light`;
  - фильтрация некорректных bbox-ов;
  - группировка аннотаций по изображениям.
- **Dataset**:
  - кастомный `RoadObjectDataset` (PyTorch `Dataset`), который возвращает `(image, target)`, где `target` содержит `boxes`, `labels`, `area`, `iscrowd`.
- **Аугментации**:
  - пайплайн `Albumentations` с корректной трансформацией и изображения, и координат bbox-ов (flip, shift/scale/rotate, color jitter и др.).
- **Сплит**: `train_test_split` с `test_size=0.2` и фиксированным seed.

### 4. Модель
- **Архитектура**: **Faster R-CNN** с backbone-ом **MobileNetV3-Large**, предобученная на COCO.
  - MobileNet -- легковесный backbone, оптимальный компромисс скорость/качество для прототипирования.
  - Голова `FastRCNNPredictor` заменяется под нужное число классов (`num_classes = 5 + 1` (фон)).
- **Стратегия обучения -- двухэтапный Transfer Learning**:
  1. **Основное обучение (10 эпох)**: backbone заморожен, обучаются только RPN и классификационная голова. Получен baseline mAP@.50 ~51-56%.
  2. **Fine-tuning (10 эпох)**: разморожены последние слои backbone-а; для них -- пониженный LR (`1e-5`), для головы -- стандартный. Итого mAP@.50 ~60%.
- **Scheduler**: `ReduceLROnPlateau` по validation loss.
- **Библиотеки**: PyTorch, torchvision (Faster R-CNN), Albumentations, torchmetrics (`MeanAveragePrecision`), OpenCV, pandas, sklearn.

### 5. Метрики
- **Основная**: **mAP@.50** (mean Average Precision при IoU > 0.5).
- **Дополнительная**: **mAP** (строгий, усреднение по IoU от 0.5 до 0.95).
- **Почему**:
  - mAP -- стандартная метрика для задач детекции (COCO benchmark);
  - @.50 -- показывает, насколько хорошо модель локализует объекты с приемлемым перекрытием;
  - строгий mAP отражает точность границ bbox-а.

### 6. Результат
- **Основное обучение** (замороженный backbone): mAP@.50 ~ **0.56**, mAP ~ **0.29**.
- **После fine-tuning**: mAP@.50 ~ **0.60**, mAP ~ **0.31**.
- Fine-tuning дал прирост ~10 п.п. по mAP@.50 по сравнению с начальным обучением.
- Модель стабильно обучалась без серьёзного переобучения; метрики на валидации уверенно росли.
- Модель сохранена (`finetuned_detector_model.pth`) и готова к интеграции.

### 7. Почему выбрана эта модель
- **Faster R-CNN** -- двухстадийный детектор, де-факто стандарт для точной детекции.
- **MobileNetV3-Large** в качестве backbone-а:
  - значительно легче ResNet/ResNeXt;
  - позволяет быстрее итерировать и прототипировать;
  - предобученный на COCO, уже "видел" автомобили, людей и т.п.
- **Двухэтапное обучение**:
  - сначала быстро получаем стабильный baseline, обучая только голову;
  - затем fine-tuning backbone-а "дожимает" качество без риска катастрофического забывания.

### 8. Сложности
- **Классовый дисбаланс**: машин в разы больше, чем пешеходов или байкеров; пришлось объединять мелкие классы.
- **Корректная аугментация bbox-ов**: при геометрических трансформациях нужно одновременно пересчитывать координаты рамок (решено через `Albumentations` с `bbox_params`).
- **Настройка двухэтапного обучения**: подбор LR для размороженных слоёв backbone-а vs. головы, чтобы не "сломать" предобученные признаки.

### 9. Улучшения

## 5. Детекция оттока клиентов (Yandex Practicum, `customer churn detection`)

### 1. Задача проекта
- **Проблема**: оператор связи "ТелеДом" хочет заранее находить клиентов, которые планируют уйти, чтобы предложить им промокоды и спецусловия.
- **Почему важно**: удержание существующих клиентов дешевле привлечения новых; проактивные удерживающие офферы напрямую снижают churn и увеличивают LTV.

### 2. Данные
- **Источник**: четыре таблицы по клиентам "ТелеДома" (contract, personal, internet, phone).
- **Тип**: табличные данные -- персональная информация, данные о тарифах, контрактах и услугах.
- **Таргет**: бинарный -- `churn` (ушёл / не ушёл).
- **Особенности**:
  - четыре отдельных CSV, которые нужно объединить по `customer_id`;
  - дисбаланс классов (ушедших меньше);
  - категориальные и числовые признаки;
  - пропуски, мультиколлинеарность некоторых фичей.

### 3. Подготовка данных
- **EDA по каждой таблице отдельно**:
  - проверка дубликатов, типов, пропусков;
  - визуализация распределений (бар-чарты, гистограммы, boxplot-ы);
  - PhiK-корреляции для категориальных и числовых признаков.
- **Feature engineering**:
  - создание таргета `churn` из контрактных данных;
  - фича `contract_dur` (длительность контракта);
  - фича `internet_services_num` (число интернет-услуг);
  - удаление мультиколлинеарных признаков.
- **Пайплайн препроцессинга**:
  - `ColumnTransformer` с `OHE`/`OrdinalEncoder` для категорий, `MinMaxScaler` для числовых, `SimpleImputer` для пропусков;
  - stratified train/test split (75/25).

### 4. Модель
- **Модели**:
  - `LogisticRegression` (GridSearch по `C`);
  - **`CatBoostClassifier`** (GridSearch по `depth`, `learning_rate`, `n_estimators`, `l2_leaf_reg`) -- финальная;
  - Soft Voting ансамбль (LightGBM + RandomForest + GradientBoosting);
  - baseline `DummyClassifier`.
- **Как работает CatBoost**:
  - градиентный бустинг на деревьях решений;
  - нативная поддержка категориальных фичей (ordered target encoding);
  - встроенная регуляризация (L2, глубина, learning rate).
- **Библиотеки**: `pandas`, `numpy`, `sklearn`, `catboost`, `lightgbm`, `matplotlib`, `seaborn`, `shap`, `phik`.

### 5. Метрики
- **Основная**: **ROC-AUC**.
- **Дополнительные**: confusion matrix, ROC-кривая, SHAP summary.
- **Почему ROC-AUC**:
  - задача бинарной классификации с дисбалансом;
  - ROC-AUC оценивает способность модели ранжировать клиентов по вероятности оттока при любом пороге;
  - не зависит от выбора конкретного threshold, что удобно на этапе разработки.

### 6. Результат
- **CatBoost**: ROC-AUC ~ **0.94** (лучшая модель).
  - Гиперпараметры после GridSearch: `depth=2`, `learning_rate=0.5`, `n_estimators=500`.
- **LogisticRegression**: ROC-AUC ~ **0.88**.
- **Soft Voting ансамбль**: качество близко к CatBoost, но сложнее в поддержке.
- **DummyClassifier** (baseline): значительно хуже.
- **Интерпретация**:
  - ROC-AUC 0.94 означает, что модель отлично разделяет "уходящих" и "остающихся" клиентов;
  - SHAP-анализ показал ключевые драйверы оттока: тип контракта (месячный), способ оплаты, число услуг, длительность контракта.

### 7. Почему выбрана эта модель
- **CatBoost**:
  - лучший ROC-AUC среди всех протестированных моделей;
  - нативная работа с категориальными фичами без ручного кодирования;
  - устойчивость к переобучению за счёт ordered boosting;
  - быстрый инференс (~0.02 с).
- LogisticRegression оставлена как fallback для real-time сценариев (0.3 с обучение).

### 8. Сложности
- **Объединение четырёх таблиц**: корректный merge по `customer_id`, обработка пропусков, проверка пересечений.
- **Мультиколлинеарность**: некоторые фичи сильно коррелируют -- нужно было убрать дублирующие признаки.
- **Интерпретация для бизнеса**: перевод SHAP-значений в конкретные рекомендации (таргетировать помесячных клиентов).

### 9. Улучшения
- Добавить **threshold tuning** и анализ бизнес-стоимости ошибок (FN дороже FP, т.к. потерянный клиент стоит больше, чем лишний промокод).
- Ввести **class_weight** или SMOTE для явного учёта дисбаланса.
- Рассчитать **доверительные интервалы** ROC-AUC через nested CV или bootstrap.
- Добавить **SHAP dependence plots** и пермутационную важность для более глубокой интерпретации.

Объединил четыре таблицы с контрактными, персональными и поведенческими данными, провёл EDA с PhiK-корреляциями, построил признаки (длительность контракта, число услуг), собрал `ColumnTransformer`-пайплайн. Сравнил логистическую регрессию, CatBoost и ансамбль с Voting. Лучшей оказалась CatBoost с ROC-AUC 0.94 после GridSearch.
